# RAG в проекте: назначение, пайплайн и метрика «процентов»

Документ для объяснения эксперту: **зачем** реализовано, **как** устроено в коде и **как считается** число, которое в UI показывается как проценты.

---

## 1. Зачем это реализовано

**Цель** — отвечать на вопросы пользователя **по содержимому его документов**, а не из общих знаний модели.

Для этого используется схема **RAG (Retrieval-Augmented Generation)**:

1. Текст документов разбивается на **чанки** (фрагменты фиксированной логики разбиения) и индексируется.
2. По тексту **вопроса** из индекса выбираются **наиболее похожие** чанки (retrieval).
3. Эти чанки подставляются в **контекст** запроса к LLM; модель формулирует ответ, опираясь на этот контекст.

Так мы ограничиваем ответ **материалом пользователя** и даём возможность показать **источники** (какой чанк из какого файла попал в контекст).

---

## 2. Как это работает по шагам (в нашем стеке)

| Этап | Где | Что происходит |
|------|-----|----------------|
| Загрузка файла | document-service | Файл сохраняется на диск, метаданные в БД. |
| Индексация | rag-service `POST /ingest` | Из файла извлекается текст (PDF/DOCX/PPTX/TXT), текст **сегментируется на чанки** (`chunking`), чанки пишутся в PostgreSQL и в **in-memory** индекс для поиска. |
| Вопрос в чате | frontend → gateway → rag-service `POST /query` | Передаётся текст запроса, `top_k`, опционально список `document_id`. Возвращаются чанки с **числовой оценкой сходства** (score). |
| Ответ пользователю | ai-service | В промпт попадают отобранные чанки + вопрос; UI дополнительно может показать те же чанки как «источники». |

**Чанк** в коде — это запись: `document_id`, порядковый `chunk_id` внутри документа, `text`, и при ответе на запрос — **`score`**.

Разбиение на чанки настраивается (`RAG_CHUNKER`, размер/overlap); по умолчанию используется **смысловое** разбиение по абзацам/предложениям (см. `rag-service` README), чтобы реже смешивать разные темы в одном фрагменте.

---

## 3. Как вычисляется релевантность (score)

В **rag-service** один общий интерфейс: для каждого чанка возвращается **вещественное число** `score`. Реализация зависит от того, включён ли **эмбеддер** (`EMBEDDER_BASE_URL`).

### Вариант A: эмбеддинги (если задан эмбеддер)

- Все тексты чанков и текст запроса переводятся в **векторы фиксированной размерности** через OpenAI-совместимый API `POST /v1/embeddings`.
- Векторы **нормируются по L2** (единичная длина).
- **Сходство** между запросом и чанком — это **скалярное произведение** нормированных векторов, то есть **косинусное сходство** между исходными векторами (диапазон для практических значений обычно **от −1 до 1**, для сонаправленных текстов чаще **0…1**).

В коде: матрица эмбеддингов чанков умножается на нормированный вектор запроса → массив `sims` по всем чанкам.

### Вариант B: TF-IDF (если эмбеддер не используется)

- По **корпусу всех чанков** строится **TF-IDF** матрица (`sklearn`: `TfidfVectorizer`, биграммы `ngram_range=(1, 2)`, до 5000 признаков).
- Запрос векторизуется **тем же векторизатором**.
- Для каждого чанка считается **косинусное сходство** между вектором запроса и строкой матрицы чанка (`sklearn.metrics.pairwise.cosine_similarity`).

Итог — снова вещественное число; для TF-IDF в фильтре отбрасываются чанки с **score ≤ 0** (неинформативное совпадение по умолчанию).

### Отбор top-k

После вычисления массива оценок:

- учитывается ограничение по `document_ids` (если задано);
- пары `(индекс чанка, score)` сортируются **по убыванию score**;
- берётся **первые `top_k`** записей.

При полном отсутствии подходящих векторов возможен **fallback**: выдача первых чанков документа с **score = 0** (технический запасной путь).

---

## 4. Откуда берётся «процент» в интерфейсе

В UI показывается не «уверенность нейросети в ответе», а **то же число `score`**, приведённое к удобному виду:

- значение **ограничивается** диапазоном **[0, 1]** (clamp);
- умножается на **100** и округляется → отображение в виде **«NN%»**.

То есть это **нормированная для отображения оценка сходства запроса и текста чанка** в выбранной модели (TF-IDF или косинус по эмбеддингам), а не вероятность правильности ответа и не метрика качества на размеченной выборке.

**Интерпретация для эксперта:**

- выше процент → текст чанка **ближе** к формулировке запроса в выбранном представлении (мешок n-грамм или эмбеддинг);
- низкий процент (например, 10–20%) означает **слабое** совпадение; чанк мог попасть в выдачу как «лучший из доступных» при малой плотности релевантного материала в индексе;
- сравнение процентов **между чанками одного ответа** обычно информативнее, чем абсолютное значение одного числа.

---

## 5. Краткое резюме одной фразой

**Реализовано:** RAG с чанковым индексом и поиском top-k по **косинусному сходству** (TF-IDF-векторы или L2-нормированные эмбеддинги). **Процент в UI** — это **score ∈ [0, 1]** после clamp, ×100, отражающий **релевантность чанка запросу на этапе retrieval**, а не качество сгенерированного ответа LLM.
